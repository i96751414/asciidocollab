## Summary

Set up a pnpm monorepo scaffold with a pure domain layer (zero external
dependencies) containing entities, value objects, repository interfaces, typed
errors, and core use cases. All domain code is tested with in-memory fakes
(not mocks). Architectural boundaries are enforced via `fresh-onion` import
validation. Code coverage thresholds at 90% for all packages except domain.

## Technical Context

**Language/Version**: TypeScript 5.5+

**Primary Dependencies**:
- `fresh-onion` вЂ” layer boundary import validation (dev dep, monorepo root)
- None for `packages/domain` (zero runtime dependencies per constitution)
- `packages/shared` вЂ” DTO types, shared TypeScript interfaces

**Storage**: N/A (Phase 1 is pure domain, no persistence)

**Testing**:
- Jest + ts-jest (all packages)
- In-memory fakes for repository interfaces (domain tests)
- `coverageThreshold` at 90% for all packages except domain
- fresh-onion as CI gate for import validation

**Target Platform**: Node.js 24.x (Active LTS "Krypton")

**Project Type**: Monorepo (pnpm workspaces) with library packages

**Performance Goals**:
- Full monorepo build: < 60 seconds (Node.js 24.x)
- Domain test suite: < 5 seconds
- fresh-onion validation: < 2 seconds

**Constraints**:
- `packages/domain`: zero runtime npm dependencies
- `packages/domain`: no decorators, no class-transformer, no DI framework
- TypeScript `strict: true` in every `tsconfig.json`
- Code coverage >= 90% for all packages except domain
- fresh-onion must pass in CI (no architectural violations)
- `Result<T, E>` discriminated union for all fallible operations
- No `any` type in production code
- No `as` type casts in production code

**Scale/Scope**: 9 entities, 18 value objects, 9 repository interfaces, 7 use
cases, 2 initial packages (`domain` + `shared`)
