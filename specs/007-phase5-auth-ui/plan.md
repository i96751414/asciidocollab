# Implementation Plan: Authentication UI & Session Flows

**Branch**: `007-phase5-auth-ui` | **Date**: 2026-05-30 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/007-phase5-auth-ui/spec.md`

## Summary

Wire the existing API auth routes into the server, add a first-run setup detection
endpoint, harden `RegisterUserUseCase` against post-setup abuse and race conditions,
fix three known bugs (cookie name, missing middleware, wrong API port), enforce
`SameSite=Strict` on the session cookie, verify CSRF protection on all POST auth
routes, and build the login and register/setup pages with full session flow —
redirect on unauthenticated access, first-run redirect to setup, post-login
navigation with open-redirect protection, and sign-out.
All new logic follows TDD with in-memory fakes for domain tests and testcontainers
for integration tests.

## Technical Context

**Language/Version**: TypeScript 6, Node.js 24+

**Primary Dependencies**:
- API: Fastify 5, `@fastify/session`, `@fastify/cookie`, `@fastify/csrf-protection`, `@prisma/adapter-pg`, argon2
- Web: Next.js 16 (App Router), React 19, Zod 4, Tailwind CSS 4, shadcn/ui

**Storage**: PostgreSQL 18 via Prisma ORM (sessions in `Session` table)

**Testing**: Jest 30 (unit + integration), Testing Library (components), Playwright (E2E)

**Target Platform**: Linux server (self-hosted) / local dev (Docker Compose)

**Project Type**: Web application — monorepo with separate API and frontend packages

**Performance Goals**: Login redirect within one navigation event; form error feedback within 1 second

**Constraints**: Edge-runtime-compatible middleware (no async API calls in middleware); no business logic in route handlers

**Scale/Scope**: Single-tenant or small-team deployment; initial focus on first-run trial experience

## Constitution Check

*GATE: Must pass before implementation. Re-checked post-design.*

### I. Clean Code
✅ All new code follows single-responsibility. `CheckSystemSetupUseCase` does one thing.
`RegistrationClosedError` is a typed domain error — no magic strings.
Route handlers delegate to use cases — no business logic in handlers.

### II. Test-Driven Development (NON-NEGOTIABLE)
✅ Every new production unit has a failing test written first:
- `RegistrationClosedError` — test before implementation
- `hasAny()` on `InMemoryUserRepository` — test before implementation
- `RegisterUserUseCase` post-setup rejection — test before adding `hasAny()` guard
- `RegisterUserUseCase` race condition handling — test before adding constraint catch
- `CheckSystemSetupUseCase` — test with empty and non-empty fake repos
- `GET /auth/setup-status` — integration test before route handler
- CSRF enforcement — integration test asserting 403 before verifying plugin config
- Login/register forms — component tests before UI implementation
- Open-redirect validation — unit test before implementing the validation helper
- Middleware — test redirect logic before implementation

### III. Seam Testing with In-Memory Fakes
✅ `UserRepository` already has `InMemoryUserRepository`. Adding `hasAny()` to the
fake before adding it to the Prisma implementation.

### Architecture — Layer Boundaries
✅ `RegistrationClosedError` and `CheckSystemSetupUseCase` live in `packages/domain`.
`GET /auth/setup-status` and `POST /auth/register` routes live in `apps/api`.
Frontend calls the API — never the database directly.

### Architecture — No Business Logic in Route Handlers
✅ All business rules (closed registration, setup status, redirect validation) live
in use cases or helpers — never in route handlers.

### Architecture — Contracts & Validation
✅ `SetupStatusDto` in `packages/shared`. Frontend forms validated with Zod.
`?redirect=` validated at the login page boundary before use.

### Security Constitution
✅ `POST /auth/register` enforced at domain level via `RegistrationClosedError`.
CSRF verified by integration tests (FR-014).
Session cookie `SameSite=Strict` enforced (FR-015).
`GET /auth/setup-status` reveals only a boolean — no user data.
All auth events logged by existing use cases.

### Technology Mandates
✅ Fastify for API, Next.js App Router for frontend, Jest + Testing Library, Playwright for E2E.
No deviation.

## Implementation Sequence (TDD — Red → Green → Refactor)

1. **[RED]** Add to `register-user.test.ts`:

2. Create `packages/domain/src/errors/registration-closed.ts`

4. **[REFACTOR]** Export `RegistrationClosedError` from `packages/domain/src/errors/index.ts`

### Step 7 — E2E Tests

Write Playwright tests covering:
1. First-run: empty database → visit `/` → redirected to `/register` → fill form → land on `/dashboard`
2. Sign in: visit `/dashboard` → redirected to `/login` → fill form → land on `/dashboard`
3. Sign out: sign in → click "Sign Out" → redirected to `/login` → `/dashboard` is inaccessible
4. Session protection: visit `/dashboard` without session → redirected to `/login`
5. Post-setup register block: after first-run setup, visit `/register` → redirected to `/login`
6. Open-redirect protection: `/login?redirect=https://evil.com` → after login, lands on `/dashboard`
