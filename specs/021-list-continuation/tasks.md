# Tasks: AsciiDoc List Auto-Continuation

**Input**: Design documents from `/specs/021-list-continuation/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/enter-command.md, quickstart.md

**Tests**: INCLUDED — the project Constitution (Principle II, NON-NEGOTIABLE) mandates
red-green-refactor TDD. Every implementation task is preceded by a failing test task.

**Organization**: Tasks are grouped by the four user stories from spec.md (US1 unordered P1,
US2 ordered P2, US3 checklist P2, US4 description P3). Each list type that the editor grammar
did not already tokenize also gets a small grammar/tokenizer addition so syntax-highlighting
stays in lockstep with continuation (research D3/D5): literal `....` block (foundational,
FR-008), explicit-number ordered `1.` (US2), dash checklists `- [ ]` (US3), and the `;;`
description separator (US4).

## Format: `[ID] [P?] [Story] Description`

- **[P]**: Can run in parallel — different file, no dependency on an incomplete task.
- **[Story]**: US1 / US2 / US3 / US4 (user-story phases only).

> **Shared-file note (drives sequencing):** All four stories extend the **same** parser,
> command, grammar, tokenizer, and test files. Tasks across different stories that touch
> these files are **NOT** parallelizable with each other and must proceed in priority order
> (US1 → US2 → US3 → US4). Within a single story, the grammar test, parser test, and command
> test are three distinct files, so those test tasks are `[P]` with each other.
>
> **`....` disambiguation:** 4+ dots followed by a space is an ordered depth-4 marker; 4+
> dots alone (line-only) is a literal block delimiter. The tokenizer branches on the char
> after the dot run (`SPACE` → ordered, `NEWLINE`/EOF → literal).

mid-content, exits on an empty item, and stays a plain newline off-list and inside verbatim
blocks — in a single undoable step. This story also builds the **shared command machinery**
(selection handling, ancestor-walking syntax-tree block suppression, single-transaction
dispatch) reused by all later stories. (No grammar change: `*`/`**`/`-` are already tokenized.)

**Independent Test**: Type `* first`, Enter → new `* ` line, cursor after marker; type
`second`, Enter twice → trailing empty bullet removed, plain blank line. `- a` continues `- `;
`** s` continues `** `.

## Notes

- `[P]` = different file, no incomplete dependency.
- All stories share `asciidoc-list-item.ts`, `asciidoc-list-continuation.ts`, the grammar +
  tokenizer + generated parser, and the three test files — hence sequential across stories.
- Grammar additions keep highlighting and continuation in lockstep: literal `....`
  (foundational), explicit `1.` (US2), dash checklist `- [ ]` (US3), `;;` description (US4).
- Highlight consistency: only `LiteralBlock` needs a **new** styleTags entry (`t.content`,
  matching `ListingBlock`); explicit `1.`, dash checklist, and `;;` reuse the **existing**
  `OrderedListItem`/`ChecklistItem`/`DescriptionList` nodes and inherit their colors — no new
  styleTags and no new `--syntax-*` var. Authoritative token colors live in `asciidoc-theme.ts`
  (applied at `Prec.highest`, so it wins over `asciidoc-highlight.ts`); tweak/verify there, not
  in `asciidoc-highlight.ts`. `asciidoc-highlight.test.ts` enforces each construct's class
  matches its existing sibling.
- FR-008 suppression walks syntax-tree **ancestors** (the marker line's direct node inside a
  verbatim block is an internal error node; the enclosing block is its ancestor).
- One transaction per Enter (FR-010) — assert single-undo and source-only change (FR-009)
  in T009; never split the dispatch in later stories.
- Out of scope (recorded in spec.md): alpha/roman ordered (`a.`, `i.`), callout lists (`<1>`).
- Commit after each green task or logical group (Constitution: commit only on green).
