# Phase 0 Research: Per-User Preview Style Preference

## R1 — Source of the Asciidoctor look (stylesheet + license)

**Decision**: Reuse the **MIT-licensed `asciidoctor-default.css`** — Asciidoctor's built-in HTML5 stylesheet — vendored verbatim to `apps/web/src/styles/vendor/asciidoctor-default.css` with its license header preserved. Treat it as a managed, re-syncable copy (do not hand-edit).

**Obtain order** (use the first that resolves):
1. Installed processor CSS: `node_modules/@asciidoctor/core/dist/css/asciidoctor.css`.
2. The `asciidoctor-stylesheets` npm package CSS.
3. The `asciidoctor/asciidoctor` GitHub repo: `data/stylesheets/asciidoctor-default.css`.

**Finding (this repo)**: The installed `asciidoctor@^3.0.4` package does **not** expose the CSS at an accessible path under `node_modules` (no `*.css` found under the package or `@asciidoctor/core`), and `asciidoctor-stylesheets` is not installed. Therefore source **(3)** — the upstream repo's `data/stylesheets/asciidoctor-default.css` (raw) — is the working source for the initial vendor. Preserve the leading MIT/comment header; record the upstream commit/tag in the vendored file's top comment for re-sync.

**Rationale**: Matching the official docs look pixel-for-pixel is unnecessary (per spec Assumptions); reusing the canonical stylesheet guarantees correct rendering of admonitions, code blocks, tables, and all list types (FR-011) without re-creating styles by hand, and the MIT license permits vendoring with attribution.

**Alternatives considered**: Hand-authoring an "Asciidoctor-like" theme (rejected — fragile, drifts from the reference, violates the "reuse OSS, do not re-create" directive); pulling the stylesheet at runtime from a CDN (rejected — adds a network dependency and a FOUC/offline risk, conflicts with FR-016).

## R2 — Scoping the vendored CSS so it cannot leak

**Decision**: Build-time selector prefixing with **`postcss-prefix-selector`** (OSS, MIT). A Node script `apps/web/scripts/build-asciidoctor-style.mjs` reads the vendored file and prefixes every rule with `.asciidoc-preview-content[data-preview-style="asciidoctor"]`, mapping root selectors (`html`, `body`, `:root`, `*`) onto the scope itself and leaving `@`-rules (`@media`, `@keyframes`, `@font-face`) intact, emitting `apps/web/src/styles/asciidoctor-style.generated.css`.

**Build wiring**: add the script to the existing `predev`/`prebuild` chain in `apps/web/package.json` (which already runs `build:worker` + `lezer-generator`). The generated file is **committed**, matching this repo's existing convention of committing generated artifacts (the lezer parser `asciidoc-parser.js` is tracked in git, regenerated by the same pre-steps). Import the generated CSS once (e.g. in `asciidoc-preview.tsx` next to `asciidoc-preview.css`) so the rules are present in the bundle whenever the attribute flips.

**Rationale**: The default Asciidoctor stylesheet uses global `html`/`body`/element selectors that WOULD restyle the whole app if injected unscoped. Build-time prefixing keeps the cascade confined to the preview content area (FR-010, SC-005) with zero runtime cost and no DOM-isolation complexity, and keeps scroll-sync's `querySelector('[data-source-line]')` working against the normal document (FR-014 untouched).

**Alternatives considered**:
- **Shadow DOM** on the preview host with unscoped CSS — kept as the documented **fallback** if PostCSS prefixing is awkward in this pipeline. Cost: scroll-sync queries in `useAsciidocPreview` must switch to read from `shadowRoot`, and `dangerouslySetInnerHTML` + DOMPurify output must be attached inside the shadow root. Chosen only if the PostCSS approach hits a wall, because it perturbs the scroll-sync seam the spec says to leave unchanged.
- **Runtime CSS-in-JS / manual nesting** — rejected; re-deriving the stylesheet defeats "reuse verbatim" and risks drift.
- **iframe** — rejected; breaks scroll-sync, shared fonts, and selection ergonomics.

## R3 — Web fonts used by the Asciidoctor look

**Decision**: Load the fonts the Asciidoctor default stylesheet expects but does **not** itself import — **Open Sans** (body), **Noto Serif** (headings/quotes), and a monospace (**Droid Sans Mono** → **Ubuntu Mono** fallback) — via `next/font/google` in `apps/web/src/app/layout.tsx`, exposed as CSS variables. The vendored stylesheet references these family names; providing them makes the Asciidoctor surface legible and faithful.

**Rationale**: `next/font/google` self-hosts the fonts at build time (no runtime third-party request, no layout shift), consistent with how `Inter`/`Urbanist` are already loaded in `layout.tsx`. Because the vendored CSS is scoped, these families only take effect inside the Asciidoctor preview surface; the app UI keeps Inter.

**Alternatives considered**: A single `<link>` to Google Fonts (acceptable but adds a render-blocking third-party request and a FOUC risk — disfavored vs. `next/font`); bundling font files manually (more maintenance, no benefit over `next/font`); relying on system fallbacks only (rejected — degrades fidelity of the reference look).

## R4 — Persistence mechanism (reuse, don't add)

**Decision**: Extend the existing **editor preferences** slice rather than introduce a new store. The field rides the same path already used by `softWrap`: `EditorPreferencesDto` (shared) → `EditorPreferences` entity + use-cases (domain) → Prisma repo (infrastructure) → `editor_preferences` table (db) → `/auth/me/editor-preferences` GET/PUT (api) → `useEditorPreferences` hook (web). The hook already does localStorage-seed + GET-on-mount + debounced PUT, which directly satisfies per-user persistence across sessions/devices (FR-007) and the no-flash seed (FR-016).

**Rationale**: The spec Assumptions explicitly say to reuse the existing per-user preference mechanism; `softWrap` is a complete, recent precedent (added as a trailing optional field with a default), minimizing risk and surface area. App color-mode lives in a separate `useTheme`/`appTheme` path and is intentionally left alone (Asciidoctor ignores it; Asciidocollab follows it).

**Alternatives considered**: A dedicated `previewStyle` endpoint/table (rejected — parallel mechanism, violates the directive and DRY); storing it only in localStorage (rejected — fails cross-device persistence in FR-007).

## R5 — Value-object vs. plain string for `previewStyle`

**Decision**: Model `previewStyle` as a validated value object **`PreviewStyle`** in `packages/domain/src/value-objects/preview-style.ts`, mirroring the existing `EditorTheme` VO (private constructor, `static parse(raw): Result<PreviewStyle, ValidationError>`, named union `'Asciidocollab' | 'asciidoctor'`).

**Divergence for FR-015**: Where the Prisma repo treats an unrecognized **theme** as fatal (throws in `toDomain`), an unrecognized stored **previewStyle** MUST fall back to the `Asciidocollab` default rather than throw — the spec requires the preview to keep rendering on a corrupt/absent value. The same default-on-invalid rule applies in the client's storage/GET parsing.

**Rationale**: A VO keeps the enum honest and consistent with the codebase's established pattern for editor theme, satisfying Clean Code (no magic strings) while the targeted fallback honors the spec's resilience requirement.

**Alternatives considered**: Plain `string` with an ad-hoc guard (rejected — inconsistent with `EditorTheme`, scatters validation); a boolean `useAsciidoctorStyle` (rejected — not extensible and reads poorly against the named two-option spec).

All Technical Context items are resolved; no `NEEDS CLARIFICATION` remain. Spec clarifications (2026-06-10) already settled: authenticated-only scope (no guest handling), no-flash initial render (FR-016), and canonical label `Asciidocollab`.
