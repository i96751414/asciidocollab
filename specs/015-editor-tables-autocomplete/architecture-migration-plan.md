## RT-001: Completion Source Cohesion Violation

### Current State

```
apps/web/src/lib/codemirror/
└── asciidoc-completions.ts   ← holds ALL completion sources
```

After spec 015, `asciidoc-completions.ts` will contain 7 distinct sources:

| Source | Trigger | Data |
|--------|---------|------|
| `attributeCompletionSource` | `{` | Document text scan |
| `xrefCompletionSource` | `<<` | Document text scan |
| `createIncludeCompletionSource` | `include::` | API file list |
| `createImageCompletionSource` | `image::`, `image:` | API file list (filtered) |
| Table skeleton snippet | `\|===` at col 0 | Static |
| Caption completion | `.` at col 0 | Static |
| Cell/row completion | `\|` at line start inside table | Static |

Each has a different trigger, data source, and test surface area. Grouping them by filename gives no useful cohesion boundary.

### Problems

- Single test file (`asciidoc-completions.test.ts`) accumulates tests for 7 unrelated triggers — growing to 300+ lines
- Any new completion type forces a developer to read the entire file to find the right place
- Violates constitution principle: "Functions MUST be small and do one thing" at module level

### Target State

```
apps/web/src/lib/codemirror/
├── asciidoc-attribute-completions.ts   ← {attr} + xref <<
├── asciidoc-path-completions.ts        ← include:: + image:: (shared factory)
├── asciidoc-table-completions.ts       ← table skeleton + cell/row snippets
├── asciidoc-caption-completions.ts     ← caption . trigger
└── asciidoc-completions.ts             ← barrel re-export only
```

```
apps/web/tests/lib/codemirror/
├── asciidoc-attribute-completions.test.ts
├── asciidoc-path-completions.test.ts
├── asciidoc-table-completions.test.ts
└── asciidoc-caption-completions.test.ts
```

### Benefits

- Each file has a clear, testable scope
- Adding a new completion source adds a new file, not lines to an existing one
- Test files match source files 1:1

## RT-002: `useEditorMount` Extension Registration Drift

### Current State

```typescript
// use-editor-mount.ts — mount effect
const state = EditorState.create({
  extensions: [
    asciidoc(), syntaxHighlighting(...), syntaxHighlighting(...),
    history(), keymap.of([...]), search(),
    readOnlyCompartment, lineNumbers(), highlightActiveLine(),
    asciidocFold, foldGutter(),
    outlineField,           ← spec 014
    showMinimap.of(...),
    autocompletion({ override: [attrib, include, xref] }),
    updateListener,
  ]
});
```

After spec 015 the list will add:
- `tableContextField` (new StateField)
- 4 more completion overrides in the `autocompletion` call
- Possibly image paths ref management

### Problems

- No stable registration boundary: every feature opens `use-editor-mount.ts` and appends
- The hook owns concerns from 6+ subsystems; constitution says "do one thing"
- New developers cannot identify what belongs in the hook vs. what is accidental scope

### Target State

```typescript
// asciidoc-editor-extensions.ts — pure factory, no React
export function createAsciiDocExtensions(config: EditorExtensionConfig): Extension[] {
  return [
    asciidoc(),
    ...syntaxExtensions(),
    ...editingExtensions(config),
    ...fieldExtensions(),              // outlineField, tableContextField
    ...completionExtensions(config),   // all completion sources
    ...uiExtensions(),                 // lineNumbers, minimap, foldGutter
    config.updateListener,
  ];
}
```

```typescript
// use-editor-mount.ts — mount only
const state = EditorState.create({
  extensions: createAsciiDocExtensions({ ... }),
});
```

### Benefits

- `useEditorMount` becomes a lifecycle manager only (mount, teardown, sync)
- New extensions are added to the factory, not the hook
- The factory is a pure function — testable without React

## Migration Phases

### Phase 1: Implement spec 015 by extending the current hook (this feature — 0 extra days)

**Goal**: Ship without scope-creep. Add `tableContextField` and image completions directly to the hook as planned in tasks.md.

**Coexistence**: The current pattern stays. The hook grows slightly.

### Phase 2: Extract `createAsciiDocExtensions` factory (next feature cycle — ~1 day)

**Goal**: Move extension list out of the hook into a testable factory.

- **Task 2.1**: Create `apps/web/src/lib/codemirror/asciidoc-editor-extensions.ts` with `createAsciiDocExtensions(config)` factory
- **Task 2.2**: Move the `extensions: [...]` array from `useEditorMount` into the factory
- **Task 2.3**: Replace the inline array in `useEditorMount` with `createAsciiDocExtensions({ ... })`

### Rollback Plan

If Phase 2 is never scheduled, the current pattern remains functional. The factory extraction is additive — nothing is broken at any intermediate state.

### Success Criteria

- [ ] `useEditorMount` contains no extension definitions (only lifecycle logic)
- [ ] `createAsciiDocExtensions` is a pure function with a unit test
- [ ] Hook public interface (`containerReference`, `viewReference`, `handleHeadingClick`) is unchanged
